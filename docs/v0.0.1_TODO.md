# CARE — Dual-mode (Ad-Hoc / Production) implementation plan

> **Scope:** turn the chat surface into a two-mode workbench.
>
> - **Ad-Hoc mode** — the chatbot generates a chain on demand, runs it
>   immediately, analyses the result, and continues an agent cycle if
>   needed (Claude-Code / ReAct-style).
> - **Production mode** — the chatbot generates a reproducible pipeline,
>   saves it to Memory, builds a quality dataset, optionally uploads to
>   a user service, and kicks off evolution on Platform.
>
> The user picks a mode via a switch right above the prompt. Mode is
> session-scoped and visible at all times.

This document is the **delivery plan**. It is not a journal — every item
either lands the feature for a real user or unblocks something that
does. Each task carries a priority (P0–P3, defined at the bottom) and
an explicit acceptance bar. Phases ship in order; tasks within a phase
can parallelise where dependencies allow.

---

## North-star user journeys

The plan exists to deliver these two journeys. Every task below should
trace back to one of them.

### Journey A — "I want an answer, fast" (Ad-Hoc)

> Ali types into chat: _"summarise the three PDFs in `~/research`
> and tell me whether they agree on the methodology"_. The chain
> generates, runs, and writes back a paragraph. If the answer needs
> a follow-up tool call, CARE loops automatically until the agent
> says it's done. Ali never sees a screen transition; the whole
> interaction is one scrollable transcript.

### Journey B — "I'm building a product" (Production)

> Maria types: _"build a weekly competitor-news digest agent"_.
> CARE generates a chain, saves it to Memory under
> `chain-weekly-competitor-news`, runs it once on today's data to
> capture a baseline, seeds a dataset card with that run, and starts
> a Platform evolution on the chain. Maria copies the chain id and
> wires it into her cron / Slack bot. A week later she comes back,
> sees the evolution converged, accepts the winner, and her bot
> auto-upgrades.

## Decisions (filled in as Phase 0 closes)

### §1 — MAGE follow-up contract (option b)

The Ad-Hoc loop drives ReAct via a **separate MAGE call** that reasons
over the previous chain's result. Rejected option (a) — pinning a
`next_action` field onto `MAGEResult` couples MAGE's schema to a
CARE-only feature and would need a coordinated upstream change.
Rejected (c) — `RequestUserInput` from CARL is for human-in-the-loop
pauses, not for "the agent decided more work was needed".

**Why (b) wins:** zero upstream changes; the second MAGE call uses
the same generator + the prior result as additional context, so the
loop is just an outer driver in `ChatScreen._run_generation`. We will
detect "done" via a sentinel pattern in the assistant-rendered
summary (`"DONE"` or `summary.is_terminal` once we add the flag).

### §2 — Dataset card shape (reuse `memory_card`)

Production-mode saves seed a `memory_card` entity per dataset entry,
tagged `dataset-entry:<chain_id>` plus `agent_run` (so the existing
RunHistory wiring picks it up). The card's `content` carries the
task / result / tokens / latency / timestamp as JSON.

**Why reuse:** keeps the change CARE-side only — no Memory schema
migration, no coordinated PR. The tag prefix gives O(1) lookups via
`memory.search_hits(tag=...)`, and the existing `record_run_completion`
helper already produces this shape. A native `dataset` entity is
revisited if/when we need per-entry indexes Memory can't synthesise
from tags.

### §3 — Loop budget defaults

| Knob                    | Default | Env override                       |
| ----------------------- | ------- | ---------------------------------- |
| Max iterations          | 5       | `CARE_CHAT__LOOP_MAX_ITER`         |
| Max tokens / session    | 50 000  | `CARE_CHAT__LOOP_MAX_TOKENS`       |
| Max wall-seconds / loop | 120     | `CARE_CHAT__LOOP_MAX_WALL_SECONDS` |

Picked to keep "fast iteration" felt instantly (≤ 5 chained MAGE
calls at typical 5s each = 25 s p99 for OpenRouter), capped at $0.50
worth of tokens at GPT-4o-mini rates, and bounded by a wall clock so a
stuck loop dies even if iteration / token guards miss.

## Phase 2 — Ad-Hoc execution path

> Goal: a single user message in Ad-Hoc generates a chain, runs it on
> the spot, streams stage events into the transcript, and surfaces
> the final answer. No persistence anywhere. Loop / ReAct lands later
> in this phase.

- [x] **DONE — Build CARL executor in `_run_generation`.** After MAGE

      constructs a fresh `ReasoningContext` via

## Phase 8 — Claude-Code-style chat polish

> Goal: bring the TUI chat surface up to Claude-Code interaction
> parity — multi-line input with hinted submission, rich Markdown +
> code-block rendering, streaming token reveal, slash-command
> autocomplete, `@`-file references, transcript search, and richer
> per-turn telemetry. Modes 0–6 nailed the *contract*; this phase
> nails the *texture*.
>
> Why now: phases 0–7 closed the dual-mode contract, but the
> surface still feels like a 2010 REPL — plain `Input`, plain
> `Static` lines, no streaming, no autocomplete, no Markdown,
> no inline cost / context indicators. Each item below converts
> one Claude-Code idiom into a CARE-native equivalent without
> changing the underlying MAGE / CARL / Memory pipelines.
>
> Ground rule: every interaction surface added here must respect
> the cross-cutting concerns below (logging, cancellation,
> idempotency, telemetry) and ship with the same test bar as
> phases 1–7.

## Cross-cutting concerns

> Things that aren't a "task" but every phase has to respect.

- **Logging.** Every state transition (mode flip, generation start /
  end, save, baseline, evolution kick) writes a record to
  `care.chat` / `care.app`. Errors are explicit `WARNING` / `ERROR`
  with `exc_info=True` (see the [extend technical logs] work already
  in place).
  - [x] **DONE — `care.chat` logger + severity-tagged mirrors.**
        Every `ChatScreen._post_line` call mirrors into the
        module-level `_log = logging.getLogger("care.chat")` so
        the full chat transcript survives in
        `logs/care-app-<ts>.log`. Default level by role: `tool`
        / stage chatter rides DEBUG (high volume); `user` /
        `assistant` / `system` lines ride INFO. Error paths
        pass `severity="warning"` / `"error"` so the mirror is
        grep-friendly alongside the raw warning logs each
        error site already emits. Every failure layer in
        `_run_generation`, `_save_chain_production`,
        `_record_production_baseline`, `_kickoff_evolution`,
        `_judge_with_rubric`, `_forget_chain_and_dataset`,
        and `/upload` calls `_log.error(..., exc_info=True)`
        before the friendly user line lands. State transitions
        (`watch_mode`, loop iteration start, generation
        start/end, save/baseline/evolution kick) all emit
        `INFO`-level records. Covered transitively by every
        existing test that asserts a `severity`-tagged
        system line — the mirror writes are the same code
        path.

- **Cancellation.** Esc must always reach the worker group. The loop
  driver in Phase 2 must check the cancellation token between
  iterations.
  - [x] **DONE — Esc cancels every long-running worker
        group.** `action_interrupt` now walks a class-level
        `_CANCELLABLE_WORKER_GROUPS` tuple (`generate`,
        `dataset`, `evolution_stream`, `forget`) and cancels
        each — empty groups no-op, so the sweep is safe to
        call unconditionally. Pre-existing UX preserved: the
        "Interrupted." visible line still gates on
        `self._generating` so the user sees the confirmation
        only when a MAGE/CARL run was actually in flight;
        other workers post their own completion / error
        messages via the `CancelledError` that propagates
        through their `await` points.

        Before this iteration Esc only cancelled the
        `generate` group, so a long-running `/dataset run`
        over 100 entries (or a live `/evolution watch`

## Out of scope (track separately)

> Things people will _ask_ for during this work but that belong in a
> future iteration. Don't let them creep in.

- Multi-user / shared sessions. CARE is single-user today; collab
  semantics need a separate design.
- Branching / forking a chat thread. Useful but orthogonal.
- Voice / multimodal input. The bottom-bar Input stays text-only.
- Self-hosted dataset eval orchestrator. We seed _one_ baseline; full
  CI-on-each-PR style eval is a `gigaevo-eval` project, not CARE.
- Auto-prompt-rewriting. We send the user's prose verbatim to MAGE in
  Ad-Hoc; we don't second-guess it.
