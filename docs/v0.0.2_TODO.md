# CARE — TUI Development TODO

> **Scope.** Prioritised, end-to-end plan to take CARE from "chat +
> settings screen, lots of draft screens" to a `uvx care`–installable
> product with the four user-facing surfaces the user actually needs:
> **Onboarding**, **Chat (current dialogue)**, **Artifacts of current
> chat**, **Saved chains library**, and **Evolution monitor**.
>
> Every CARE-side task is reachable via a slash command (`/onboard`,
> `/artifacts`, `/library`, `/evolution`, …) so the chat screen stays
> the single entry point.
>
> Cross-module asks (gigaevo-memory, gigaevo-platform, carl-mage,
> carl-experiments) are inlined under the CARE feature that needs them
> at a strictly lower priority — that way one list shows everything
> that has to happen for the feature to feel complete.
>
> **Priority labels appear at the start of every task** as `[P0]` …
> `[P4]`. P0 ships M0 (uvx-installable, four screens reachable, happy
> path works). P1 fills gaps. P2 polish. P3 stretch. P4 exploratory.
>
> **Today's reality.** `care/screens/` has ~30 screen files. The two
> screens the user identifies as "main" — `chat.py` (~11k LoC) and
> `settings.py` (~420 LoC) — are real surfaces. Others
> (`library.py`, `evolution.py`, `inspection.py`, `marketplace.py`,
> `catalog.py`, …) are drafts: they compose, render placeholder data,
> and aren't wired to a slash command or to a finished data layer.
> Their existence is leverage — most P0 work below is **finishing**
> a draft, not building from zero.

---

## §0 — Release shape & ground rules

- **Distribution.** `uvx care` is the canonical install. `pyproject.toml`
  already declares `care = "care.cli:main"` and lazy-imports every
  upstream — so `pipx run care` / `uvx care` boots even when the user
  has nothing configured. The onboarding wizard (§1) finishes the
  picture.
- **First-boot heuristic.** Today `CareApp.__init__` flips
  `mode = "first_run"` when `~/.config/care/config.toml` is missing.
  That's the trigger for the onboarding flow.
- **Slash commands.** Every navigable screen below MUST be reachable
  by a `/<command>` typed into the chat input. Chat is the home base
  — modal screens are pushed via the command, dismissed via `Esc`.
- **Cross-module rules.** Items marked `[→ <repo>]` belong to a
  sibling repo (`gigaevo-memory`, `gigaevo-platform`,
  `gigaevo-memory/client/python`, `carl-mage`, `carl-experiments`).
  They sit one priority lower than the CARE work that consumes them,
  so CARE can ship a mocked / disabled-feature version first.

generates during the current dialogue is currently in-memory only;
the user has no way to see, diff, or persist them.

that the user might want to save: generated CARL chains (primary),
intermediate MAGE stage payloads (already projected by
`care/intermediate_artifacts.py`), tool / skill outputs from Ad-Hoc
execution, dataset eval rows (Production mode), LLM follow-up
answers worth keeping.
