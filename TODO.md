### QA

- [ ] Textual ships a headless test harness. run_test() runs your app in "headless" mode (no output) and drives the app via a Pilot object, and Pilot can simulate key-presses, clicks, double/triple clicks, and hovering at a specified position. Because it never grabs the real terminal, Claude Code gets a clean text-in/text-out loop: drive the app, inspect the actual widget tree, print results. The single most important gotcha: some actions won't change state immediately — messages take a moment to bubble, so if you assert immediately it may fail because the message hasn't been processed; calling pause() waits for all pending messages. Put await pilot.pause() after every interaction. For background work, await app.workers.wait_for_complete() waits for workers to finish.
- - ```
    # drive.py — run with: uv run drive.py
    import asyncio
    from my_app import MyApp

    async def main():
        app = MyApp()
        async with app.run_test(size=(120, 40)) as pilot:
            await pilot.press("tab", "enter")     # keyboard navigation
            await pilot.pause()                   # let messages settle
            await pilot.click("#submit")          # click by CSS selector
            await pilot.pause()
            status = app.query_one("#status")     # inspect real state
            print("STATUS:", status.renderable)   # prints fine — app is headless
            app.save_screenshot("state.svg")      # capture current screen as SVG
    asyncio.run(main())
    ```

### Performance & smoothness (Textual TUI)

_From a rendering/runtime audit of the `care` TUI. Ordered by impact. The single biggest smoothness risk is the unbounded chat transcript (P-1); the O(n²) toggle walks (P-2) and blocking file extraction (P-3) are the most visible "the app froze" moments. Each task names a real `file:line` to start from. Add a Pilot regression (P-9) so these don't regress._

- [x] **P-1 — Cap / virtualize the chat transcript widget tree (HIGH).** `_post_line` (`care/screens/chat.py:13280`, mount at `:13435`) mounts one widget per line into `#chat-transcript` with **no upper bound**; `self._lines` *and* the mounted DOM grow unbounded for the whole session, and every assistant/system line is a heavy `Markdown` widget. After a few hundred lines, every `query()` walks a huge tree, layout recalcs get expensive, and memory climbs linearly. Fix: keep the full `self._lines` model (search / transcript export / `relocalize` depend on it) but bound the number of *mounted* widgets to a `MAX_RENDERED_LINES` (~300–500). Unmount the oldest widgets past the cap behind a `▲ N earlier lines — press X to load` affordance and re-mount on demand. This is the highest-leverage change for long sessions.
- [x] **P-2 — Kill the O(n²) full-transcript walks on toggles (HIGH).** `action_toggle_compact_mode` (`care/screens/chat.py:11841`) and the step-body collapse toggle (`~:11811`) loop over **every** line and call `query_one("#chat-line-{idx}")` *per line* — each a full O(n) DOM walk, so the whole loop is O(n²) — then re-parse each `Markdown` widget. On a long session, Ctrl+D / Ctrl+E freeze the UI for seconds. Fix: (a) resolve widgets once via a single `transcript.query(".chat-line")` (or a cached `id→widget` map) instead of per-line `query_one`; (b) only re-render lines whose projection actually changed; (c) chunk via `call_after_refresh` (or a worker) so the UI stays responsive. The same per-line-`query_one` pattern is in the search-highlight pass — fix it there too.
- [x] **P-3 — Move `@<path>` file/PDF/Office extraction off the UI thread (HIGH).** `_resolve_file_refs(task)` runs **synchronously** in `_handle_task` at `care/screens/chat.py:3034`, *before* the generation worker spawns (`:3069`). A multi-MB PDF (`pypdf`) or `.docx`/`.pptx`/`.xlsx` extraction blocks the event loop — the entire UI freezes, can't be cancelled, shows no progress. Fix: do ref resolution inside the worker (or a dedicated `asyncio.to_thread`), and emit `▶ Reading @file…` / `✓` tool lines so the user sees progress and can Esc-cancel.
- [x] **P-4 — Debounce keystroke-driven work (MED).** `on_input_changed` (`care/screens/chat.py:12372`) re-runs the **full** search-highlight pass (`_apply_search_query`, which walks every line — see P-2) on *every* keystroke in `#chat-search-input`, and fires `_refresh_autocomplete` + `_refresh_input_hints` on every `#chat-input` keystroke. Add a ~120–150 ms debounce (reset a `Timer` on each change) so a fast typist triggers one pass per pause, not one per character.
- [x] **P-5 — Diff DataTable rows instead of `clear()` + re-add on every poll (HIGH).** `EvolutionDashboard._apply_view` (`care/screens/evolution_dashboard.py:705`) calls `table.clear()` then re-adds every row each ~10 s tick (`:712`), even though rows already carry a stable `key=row.run_id`. This wipes the user's selection + scroll position and is O(rows) churn per tick. Switch to incremental reconciliation: `update_cell(key, col, val)` for changed cells, `add_row(..., key=)` for new run_ids, `remove_row(key)` for gone ones. The identical `clear()`-then-re-add pattern recurs in `artifacts.py` (`_apply_rows`, rebuilds *all* tab tables), `library.py`, `runs.py`, `cost.py`, `lineage.py`, `evolution.py` — fix the dashboard first as the template, then sweep the rest.
- [x] **P-6 — Make polling timers adaptive / non-stacking (MED).** Inventory: StatusBar 5 s (`care/widgets/status_bar.py:138`, probes Memory + Platform health), evolution footer 3 s (`care/screens/chat.py:1612`, `platform.running_evolution_count` via `asyncio.to_thread`), dashboard 10 s full refetch (`evolution_dashboard.py:575`), inspection freshness 30 s (`inspection.py:438`). Two cheap wins: (a) back the cadence off — or stop entirely — when the screen is unfocused/backgrounded or nothing is in flight (e.g. no running evolutions ⇒ stop the footer poll); (b) confirm every polling worker is `exclusive=True` so a slow backend round-trip can't stack workers.
- [x] **P-7 — Stop idle spinner repaints (LOW).** `ExecutionScreen` arms a 0.1 s `_tick_spinner` (`care/screens/execution.py:241`) that keeps firing 10×/s even after all steps finish (the callback early-returns, but the timer lives on). Stop/standby the interval when no step is `running` — mirror the chat status-anim pattern (`care/screens/chat.py:7317` `_sync_status_anim_timer`, which already pauses its timer when idle). Same idle-guard for `evolution_compare.py:408`.
- [x] **P-8 — Use native loading state for fetch screens (LOW, also a UX win — see A-7).** `library` / `inspection` / `evolution` / `artifacts` track a `loading` bool and paint a *static* "loading…" placeholder. Textual's `widget.loading = True` gives a free animated, reduced-motion-aware overlay during the fetch. Swap the placeholders for it.
- [x] **P-9 — Add a headless perf/smoke harness (regression guard for P-1…P-4).** Reuse the QA `run_test()` Pilot pattern above: a `drive.py` that posts ~1000 chat lines, toggles compact mode, and runs a search, asserting (a) the mounted-widget count stays ≤ the P-1 cap and (b) each interaction completes under a wall-clock budget. Locks in the fixes so they don't silently regress.

### Animations & motion (UX)

_The TUI currently has **zero** `.animate()` calls and **zero** CSS `transition:` rules — every state change is an instant pop, and the only motion is hand-rolled `set_interval` spinner ticks that ignore the user's motion preference. Adopt Textual's native animation APIs (`widget.styles.animate(...)` + CSS `transition:`), which **auto-no-op when `TEXTUAL_ANIMATIONS=none` / the app animation level is reduced**, so reduced-motion comes for free. Ordered by ROI. Note the interaction with the perf section: animate cheaply and **skip bulk mounts** (A-1), or you'll fight P-1/P-2._

- [x] **A-0 — Foundation: standardize on native APIs + a reduced-motion switch.** Replace ad-hoc motion thinking with Textual's `styles.animate()` / CSS `transition:`. Expose a `CARE_*` config (`defaults.reduced_motion`) wired to Textual's animation level / `TEXTUAL_ANIMATIONS`, so SSH and slow-terminal users can opt out and every animation below honors it automatically. This is the umbrella for A-1…A-8.
- [x] **A-1 — Fade/slide-in new chat lines (HIGH ROI).** `_post_line` (mount at `care/screens/chat.py:13435`) pops every line in instantly. After mount: set `widget.styles.opacity = 0`, then `widget.styles.animate("opacity", 1.0, duration=0.18, easing="out_cubic")` (optionally a 1-cell `offset` rise). Keep it to a **single cheap opacity tween — no layout thrash** — and **skip it during bulk mounts** (session restore, `/replay`, branch switch, transcript reload), where dozens of lines mount at once and per-line animation would stutter and fight P-1. Gate on A-0.
- [x] **A-2 — Modal entrance fade/scale (HIGH ROI, pervasive).** The modals (`confirm.py`, `help_modal.py`, `command_palette.py`, `file_picker.py`, `save_chain_name.py`, `save_agent.py`, `save_report.py`) center-pop with no easing. In each modal's `on_mount`, start the box at `opacity 0` and `.animate("opacity", 1.0, 0.15, easing="out_cubic")` (optionally animate `offset`/height for a subtle rise). Factor a small `AnimatedModalScreen` base so all share one implementation. (Don't depend on a `ModalScreen.TRANSITION` attribute — verify the exact Textual 8.x API first; the `on_mount` + `.animate()` path is guaranteed.)
- [x] **A-3 — Toast in/out fade (MED ROI).** `ToastHost` mounts rows (`care/widgets/toast.py:158`) and `_dismiss` removes them (`:169`) instantly. Fade in on `push` (start `opacity 0` → animate to 1 over ~0.25 s); on dismiss, `.animate("opacity", 0, 0.2, on_complete=row.remove)` instead of removing outright. Optional tiny stagger (`delay=0.04*index`) when several stack.
- [x] **A-4 — Smooth the stage-trail muting + pipeline-strip reveal (MED ROI).** When a MAGE stage completes, `_mark_stage_started_line_done` (`care/screens/chat.py:~10586`/`~10613`) instantly swaps the `▶ stage` line (yellow) → grey + strike via the `chat-line-stage-done` class (`_STAGE_DONE_CSS_CLASS`, `:10559`). Add `transition: color 0.3s out_cubic;` to `.chat-line-tool` so the mute is a smooth grey-wash. For the pipeline strip show/hide (`:7312`, currently a hard `display` flip), animate `opacity` (and height) instead of toggling `display` so it slides in/out.
- [x] **A-5 — Trivial CSS transitions on highlights, buttons, toggle (MED/LOW, ~1 line each).** Add `transition: background 0.15s;` to `.chat-line-search-match` / `.chat-line-search-current` (`care/screens/chat.py:656`) so Ctrl+F cycling fades instead of snapping; `transition: background 0.15s, color 0.15s;` to `#chat-stop-btn` (`~:734`) for smoother hover/active feedback; and to the mode-toggle `.toggle--button` (`~:720`) for when the mode row is unhidden. CSS-only, no Python.
- [x] **A-6 — Animate auto-scroll on new lines, conditionally (LOW).** Every `_post_line` calls `transcript.scroll_end(animate=False)` (`:13436`, also `:1756/:1770/:7975/:8202/:10519`). When the user is already pinned to the bottom, `animate=True` gives a smooth follow; keep it **instant during bulk mounts and when the user has scrolled up** (never yank them). Low-risk once A-1's bulk-mount guard exists.
- [x] **A-7 — Native `LoadingIndicator` / `widget.loading` for fetch states (LOW — same change as P-8).** Replace the static "loading…" placeholders in `library`/`inspection`/`evolution`/`artifacts` with Textual's animated `loading` overlay: consistent, animated, and reduced-motion-aware out of the box.
- [x] **A-8 — Welcome splash fade (LOW, boot-only).** `WelcomeScreen` shows the MAESTRO splash for 200 ms of dead air then hard-switches (`care/screens/welcome.py:162`). Add a quick fade-in on mount + fade-out before `_route()`. Lowest priority.
- [x] **A-9 — (Non-goal) leave the existing tick-spinners alone unless routed through A-0.** The thinking pulse (`care/screens/chat.py:7220`–`7348`) and braille step spinners (`execution.py`, `evolution_compare.py`) already read well — the only reason to touch them is so `reduced_motion` (A-0) can freeze them (they currently ignore motion prefs). Don't rewrite them into `.animate()` for its own sake.
